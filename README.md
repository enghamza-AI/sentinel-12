# ‚òÑÔ∏è sentinel-12

> **Detect hazardous asteroids. Minimize misses. Understand why accuracy lies.**

A hands-on ML learning project built around **confusion matrices, recall optimization, and the precision-recall trade-off** ‚Äî using the lens of a planetary defense early-warning system.

---

## What I'll learn

This project is not about building the best asteroid detector. It is about learning to **evaluate any classifier correctly** ‚Äî a skill more valuable than picking a fancier model.

By the time I've run through this project I will understand:

- Why a model with **94% accuracy** can be completely useless
- What the **confusion matrix** is and how to read every cell (TP, TN, FP, FN)
- The difference between **precision** and **recall**, and when each one matters
- Why in high-stakes systems you **optimize for recall**, not accuracy
- How **decision threshold tuning** lets you control the precision-recall trade-off without retraining
- What the **ROC curve and AUC** tell you (and when the PR curve is more honest)
- How **class imbalance** silently corrupts training if you don't handle it
- What **data leakage** is and how to avoid it

Read [`ABOUT_THE_PROJECT.md`](ABOUT_THE_PROJECT.md) for a complete explanation of every concept ‚Äî it's written as a study companion, not as documentation.

## The core idea in 60 seconds

We have 6,000 simulated asteroids. Only **6% are actually hazardous** ‚Äî the rest are safe. We train a model and ask: is it any good?

A naive answer: "Our model is **93% accurate**! Ship it."

A smarter answer: look at the confusion matrix.

```
                    Predicted SAFE   Predicted HAZARDOUS
  Actually SAFE         TN=1326         FP=84       <- false alarms
  Actually HAZARDOUS    FN=18           TP=72        <- FN = MISSED THREATS üö®
```

That `FN=18` means **18 real hazards** were called "safe" and slipped through. Meanwhile, the dumbest possible model ‚Äî one that just predicts "safe" for everything ‚Äî scores **94% accuracy** while catching exactly zero hazards. Accuracy is a lie here.

The fix: optimize for **recall** (`TP / (TP + FN)`), which measures how many real hazards we actually caught. By tuning the decision threshold from 0.5 down to 0.35, we push recall from 80% to **90%**, cutting missed threats nearly in half:

```
                    Predicted SAFE   Predicted HAZARDOUS
  Actually SAFE         TN=1282         FP=128       <- more false alarms (acceptable)
  Actually HAZARDOUS    FN=9            TP=81         <- fewer missed threats ‚úÖ
```

That trade-off ‚Äî more false alarms in exchange for catching more real threats ‚Äî is the central lesson.

## Key results summary

| Metric | Default threshold (0.5) | Tuned threshold (0.35) |
|---|---|---|
| Accuracy | 93.2% | 90.9% |
| **Recall** | 80.0% | **90.0%** ‚Üê the goal |
| Precision | 46.2% | 38.8% |
| F1 | 0.585 | 0.542 |
| False Negatives (missed threats) | 18 | 9 |
| False Positives (false alarms) | 84 | 128 |
| AUC (ROC) | 0.952 | 0.952 (threshold-independent) |
